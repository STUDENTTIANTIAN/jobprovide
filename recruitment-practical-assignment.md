# 开发岗位实战测试题（Demo 交付版）

## 一、测试目的

本测试用于考察候选人的实际学习能力、问题拆解能力、工程实现能力和交付意识，不以算法背诵或代码行数作为主要评价依据。

请在以下两道题中任选一题完成；如招聘负责人指定题目，则以指定题目为准。

允许使用 AI 辅助开发，但候选人必须能够解释关键方案、独立排查问题，并说明 AI 在项目中的使用范围。

## 二、统一交付要求

候选人需要提交以下内容：

1. 可访问的公网演示地址，并提供测试账号（如需要登录）。
2. 完整源代码仓库地址。
3. 项目 README，至少包含启动方法、部署方式、架构说明和测试步骤。
4. 一段 3～5 分钟的演示视频，展示完整业务流程。
5. 已知问题、未完成事项以及后续优化方向。
6. 面试现场能够演示部署结果，并回答实现思路和故障处理问题。

### 基本要求

- 演示环境应当可以由面试官独立操作，不能只提供截图或录屏。
- 页面应具有基本可用性，不能依赖开发者手动修改数据库才能完成流程。
- 关键数据需要持久化，刷新页面后不能全部丢失。
- 对加载中、空数据、执行失败和重复操作提供明确反馈。
- 不得提交公司现有系统代码、私有仓库代码或来源不明的商业代码。
- 不要求使用指定技术栈，候选人可以选择自己熟悉的方案。

## 三、题目一：手机 Agent 闹钟任务调度系统

### 1. 业务背景

开发一个手机 Agent 系统。管理人员在网页后台创建任务，Android Agent 接收任务后，自动操作模拟器自带的“时钟/闹钟”App，并将执行结果返回后台。

该题不需要真实安卓手机，统一使用 Android 模拟器验收。

### 2. 标准测试环境

- Android Studio Emulator
- Android 12 / API 31
- Google APIs `x86_64` 镜像
- 分辨率建议为 `1080 × 1920`
- 模拟器不 Root
- 允许安装 APK，并人工开启一次无障碍服务权限
- APK 应支持 `x86_64`，推荐提供 Universal APK

如候选人使用其他环境开发，最终仍需保证在上述标准环境中可以完成验收。

### 3. 必须实现的功能

#### 管理后台

1. 展示已注册的模拟器设备。
2. 展示设备在线、离线和最后心跳时间。
3. 可以选择设备并创建闹钟任务。
4. 任务至少包含闹钟时间、闹钟名称和是否开启振动。
5. 展示任务的等待执行、执行中、执行成功和执行失败状态。
6. 展示 Agent 上传的执行日志、结构化结果和结果截图。

任务示例：

```json
{
  "time": "08:35",
  "label": "项目晨会",
  "vibrate": true
}
```

#### Android Agent

1. APK 安装后能够向后台注册设备并获得设备标识。
2. 定时发送心跳，让后台识别设备在线状态。
3. 接收后台下发的闹钟任务。
4. 自动打开系统时钟 App，并进入闹钟页面。
5. 根据任务设置闹钟时间、名称和振动选项。
6. 保存后确认页面中出现正确的闹钟。
7. 将执行结果、过程日志和结果截图上传到后台。
8. 任务执行失败时返回明确的失败原因。

### 4. 关键限制

- 必须真实操作时钟 App 的可见界面。
- 不允许通过修改时钟数据库直接创建闹钟。
- 不允许使用 `adb shell am`、`adb input` 或外部脚本代替 APK 完成核心操作。
- 可以使用 Android 无障碍服务、UIAutomator、视觉识别或候选人自主设计的界面操作方案。
- 面试时会随机指定闹钟时间和名称，不能将测试数据写死在代码中。

### 5. 异常场景要求

至少处理以下场景中的两项：

- 网络短暂断开后自动恢复并继续上报。
- 后台重复下发同一任务时，不重复创建相同闹钟。
- Agent 被关闭并重新启动后，可以恢复未完成任务。
- 页面结构发生轻微变化时，能够给出可理解的失败信息。
- 任务超时后停止执行，避免 Agent 长时间卡死。

### 6. 加分项

- 支持修改、启用、停用和删除指定闹钟。
- 后台可以实时查看任务执行步骤。
- 支持任务取消和失败重试。
- 对不同分辨率或 Android 版本具有一定兼容性。
- 具有完整的自动化测试或可重复执行的验收脚本。

### 7. 现场验收流程

1. 在全新的标准模拟器中安装候选人提交的 APK。
2. 开启必要权限，确认设备在后台显示为在线。
3. 面试官随机创建一个闹钟任务。
4. 观察 Agent 是否真实操作时钟 App。
5. 对比闹钟时间、名称和振动设置是否正确。
6. 检查后台日志、执行结果和截图。
7. 模拟一次断网、重复任务或 Agent 重启场景。

## 四、题目二：AI 字幕分析与素材匹配平台

### 1. 业务背景

开发一个用于视频制作的轻量化平台。用户上传口播视频、音频或字幕文本后，系统对字幕进行整理和语义分段，再从素材库中推荐与字幕内容匹配的图片或视频素材。

该题重点不是制作一个完整的视频剪辑软件，而是完成“内容输入、字幕处理、素材匹配、人工调整、结果保存”的业务闭环。

### 2. 必须实现的功能

#### 内容输入与任务处理

1. 支持上传口播视频、音频，或者直接粘贴字幕文本。
2. 上传视频或音频后，可以通过语音识别获得字幕。
3. 耗时操作应以异步任务执行，并展示处理进度和失败状态。
4. 任务失败后可以查看原因并重新执行。

如果候选人没有可用的语音识别服务，可以同时保留“粘贴字幕文本”作为稳定演示入口，但需要说明真实语音识别的接入方案。

#### 字幕整理

1. 展示原始字幕内容。
2. 按句子或语义将字幕划分为多个片段。
3. 每个片段可以人工修改文本和顺序。
4. 展示每个字幕片段的关键词或主题。

#### 素材库

1. 支持上传图片或短视频素材。
2. 支持为素材填写名称、标签和关键词。
3. 至少准备 10 个可用于演示的素材。
4. 支持按关键词搜索或筛选素材。

#### 智能匹配

1. 为每个字幕片段推荐相关素材。
2. 每个字幕片段至少展示 3 个候选素材。
3. 展示匹配理由或匹配关键词，不能只给出无法解释的分数。
4. 用户可以人工替换最终采用的素材。
5. 保存字幕片段与最终素材的对应关系，刷新后仍能查看。

### 3. AI 使用要求

- 可以使用大语言模型、Embedding、关键词匹配或混合方案。
- 需要在 README 中说明所采用的匹配策略以及选择该策略的原因。
- 外部 AI 服务失败时，应提供错误提示；提供规则降级方案可获得加分。
- 不得在前端代码或公开仓库中直接暴露 API Key。
- 面试时会提供一段新的字幕，系统不能只对演示样例有效。

### 4. 加分项

- 自动为新素材生成标签和关键词。
- 使用向量相似度结合关键词进行混合排序。
- 支持字幕片段的拖动排序和素材快速替换。
- 支持视频时间轴或简单的组合预览。
- 可以生成带匹配素材的预览视频。
- 对重复任务、超时任务和并发任务具有可靠处理机制。
- 记录每次 AI 匹配的模型、输入版本和结果，便于追溯。

### 5. 现场验收流程

1. 面试官上传一段新的口播音频、视频或字幕文本。
2. 查看任务状态以及字幕生成或解析结果。
3. 检查字幕语义分段是否基本合理。
4. 检查每个片段推荐的素材和匹配理由。
5. 搜索一个关键词并人工替换某个字幕片段的素材。
6. 刷新页面，确认人工选择和任务结果仍然存在。
7. 模拟一次 AI 服务失败或处理失败，检查错误提示和重试能力。

## 五、统一评分标准

总分 100 分：

| 评分维度 | 分值 | 主要观察内容 |
| --- | ---: | --- |
| 业务闭环 | 30 | 核心流程是否真实可用，能否从输入走到最终结果 |
| 稳定性 | 20 | 异常提示、重试、幂等、持久化和边界场景处理 |
| 工程质量 | 15 | 代码结构、接口设计、配置管理和基本安全意识 |
| 学习与解决问题能力 | 15 | 面对陌生领域的方案选择、验证过程和问题复盘 |
| 产品体验 | 10 | 页面信息是否清晰，操作是否连贯，状态反馈是否明确 |
| 部署与文档 | 10 | 公网环境是否可访问，README 和演示说明是否完整 |

### 评价原则

- 核心闭环可用，比堆叠大量未完成的功能更重要。
- 能解释为什么这样设计，比单纯调用现成代码更重要。
- 能清楚说明不足和改进方法，不会因为存在合理的未完成项被直接否定。
- 出现问题时的定位过程，是面试评价的重要部分。
